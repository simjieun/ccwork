# Content & Tone

문서 사이트의 모든 UI 카피는 **concise, confident, implementation-focused** 톤을 따른다.

## 원칙

1. **간결**: 한 문장에 한 가지 의미만. 부가 설명은 다음 줄로 분리한다.
2. **단정**: "할 수 있을지도 모릅니다" 같은 약한 표현을 피한다. "한다 / 하지 않는다"로 단정한다.
3. **구현 지향**: 사용자 행동·코드 결과에 직결되는 문장을 쓴다.
4. **시스템 일관**: 같은 행동에는 같은 단어를 쓴다. 예: "저장(Save)"을 "보관"으로 바꿔 쓰지 않는다.

## 라벨링 규칙

- 버튼 텍스트는 **동사 + (명사)** 형태로 쓴다 — 예: `Save`, `Create note`.
- 단순한 확인 버튼은 `OK` 대신 행동 동사를 쓴다(예: `Delete`, `Apply`).
- 토글류는 상태가 아닌 행동으로 라벨링한다 — 예: `Enable notifications` (현재 켜졌다는 의미 아님).
- 링크는 목적지를 드러낸다 — `여기 클릭` ❌, `대시보드로 이동` ✅.

## 에러·상태 메시지

- 무엇이 잘못됐는지 + 어떻게 해결하는지 두 부분으로 작성한다.
- 사용자 잘못이 아닌 시스템 오류는 사과보다는 다음 행동을 안내한다.

| 상황         | ❌ Don't               | ✅ Do                                              |
| ------------ | ---------------------- | -------------------------------------------------- |
| 저장 실패    | "오류가 발생했습니다." | "저장에 실패했습니다. 잠시 후 다시 시도해 주세요." |
| 빈 인풋 제출 | "필수 입력입니다."     | "제목을 입력해 주세요."                            |
| 권한 부족    | "권한 없음"            | "이 작업에는 관리자 권한이 필요합니다."            |
| 로딩 안내    | "로딩…"                | "노트를 불러오는 중…"                              |

## 빈 상태(Empty state)

- 무엇이 비어있는지 한 줄 + 다음 행동(액션 버튼)을 함께 둔다.
- 빈 상태 자체를 부정적으로 표현하지 않는다 — "아직 노트가 없습니다" + `Create note` 버튼.

## Microcopy 예시 (동작 동사 통일)

| 카테고리 | 권장                | 비권장                 |
| -------- | ------------------- | ---------------------- |
| 생성     | `Create`, `New ___` | `Add new ___`, `Make`  |
| 저장     | `Save`              | `Keep`, `Store`        |
| 삭제     | `Delete`            | `Remove`, `Erase`      |
| 확인     | `Confirm`, `Apply`  | `OK`, `Yes`            |
| 닫기     | `Close`, `Cancel`   | `Dismiss`, `Nevermind` |

## Do / Don't

**Do**

- 한 행동에는 한 단어를 정해 끝까지 같은 단어를 쓴다.
- 사용자가 다음에 무엇을 할지를 문장에 포함한다.
- 숫자·기간 등 구체값은 정확하게 표시한다.

**Don't**

- "혹시", "아마", "조금" 같은 약한 부사를 쓰지 않는다.
- 의미가 모호한 라벨(`Submit`, `Go`, `Click here`)을 쓰지 않는다.
- 사과·이모지로 본질을 덮지 않는다.
