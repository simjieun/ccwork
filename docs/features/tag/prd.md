# 태그 기능 PRD

> 관련 문서: [`spec-fixed.md`](./spec-fixed.md), [`spec-original.md`](./spec-original.md)

## 1. 개요

노트 앱에 **태그(tag)** 개념을 도입한다. 사용자는 노트 상세 화면에서 자유 텍스트로 태그를 입력해 chip 형태로 확정하고, 불필요한 태그는 chip 단위로 삭제할 수 있다. 태그는 별도 엔티티가 아니라 각 노트에 임베드된 문자열 배열(`tags: string[]`)로 저장되며, 제목·본문과 함께 저장 버튼을 눌렀을 때 일괄로 PATCH/POST 된다.

본 기능의 목적은 다음과 같다.

- 노트를 주제·맥락별로 분류할 수 있는 가장 가벼운 메타데이터를 제공한다.
- 추후 필터/검색/자동완성 등 태그 기반 기능 확장을 위한 데이터 기반을 마련한다.
- 기존 노트 CRUD 흐름과 저장 패턴을 깨지 않는 선에서 점진적으로 도입한다.

성공 기준(이 단계에서):

- 노트 상세 화면에서 태그를 추가·삭제할 수 있다.
- 저장 시 `tags`가 노트와 함께 영속화된다.
- 기존 노트(레거시 데이터)도 태그 필드 없이/빈 배열로 안전하게 렌더링된다.

## 2. 사용자 스토리

### US-1. 태그 추가

> **As a** 노트 작성자
> **I want to** 노트 상세 화면에서 자유 텍스트로 태그를 입력하고 Enter나 쉼표로 확정할 수 있다
> **So that** 노트를 주제별로 가볍게 분류할 수 있다.

- 입력값은 trim 되고, 빈 문자열이면 조용히 무시된다.
- 1~20자만 입력할 수 있다(`maxLength=20`).
- 대소문자 무시 기준으로 이미 존재하는 태그면 추가되지 않는다.

### US-2. 태그 삭제

> **As a** 노트 작성자
> **I want to** chip의 `✕` 버튼을 클릭하거나, 빈 input에서 Backspace를 눌러 태그를 제거할 수 있다
> **So that** 잘못 추가했거나 더 이상 의미 없는 태그를 빠르게 정리할 수 있다.

### US-3. 태그 표시

> **As a** 노트 열람자
> **I want to** 노트 상세 화면에서 현재 노트에 달린 태그 목록을 chip 형태로 한눈에 볼 수 있다
> **So that** 노트의 분류·맥락을 즉시 파악할 수 있다.

### US-4. 저장과 일괄 반영

> **As a** 노트 작성자
> **I want to** 제목·본문·태그를 한 번의 저장 동작으로 함께 영속화할 수 있다
> **So that** chip을 만질 때마다 네트워크 요청이 발생하지 않고, 저장 시점이 예측 가능하다.

### US-5. 한글/IME 입력 안전성

> **As a** 한글 사용자
> **I want to** IME 조합 중 Enter가 chip 확정으로 오작동하지 않는다
> **So that** 한글 태그를 의도대로 입력할 수 있다.

### US-6. 레거시 노트 호환

> **As a** 기존 노트를 가진 사용자
> **I want to** 태그 필드가 없는 기존 노트도 에러 없이 열람·편집할 수 있다
> **So that** 마이그레이션 과정에서 데이터가 깨지지 않는다.

## 3. 기술 결정 (ADR)

### 3.1 컴포넌트 아키텍처 — `useNoteEditor` 커스텀 훅 + 합성

**Context** — 태그 기능이 들어오면 `NoteEditor`는 기존 `title`/`content`/`saving` 외에 `tags` state, 정규화 규칙(trim·빈값 거부·길이·대소문자 중복), 일괄 저장 분기까지 떠안게 된다. spec §5.4는 `TagInput`을 별도 하위 컴포넌트로 두라고 명시하지만, 정규화 로직을 어디에 둘지(컴포넌트 내부 / 훅 / 순수 함수)와 폼 state를 누가 보유할지(컴포넌트 / 훅)는 열려 있다. 또한 본 프로젝트는 향후에도 합성 기반·커스텀 훅 활용을 확장 패턴으로 가져갈 방침이고, 정규화 같은 순수 로직은 RTL 없이 단위 테스트로 검증되길 원한다.

**Decision** —

- `NoteEditor`의 폼 상태와 저장 로직(`title`/`content`/`tags`/`saving` + `handleSave`)을 **`useNoteEditor` 커스텀 훅**으로 추출한다. `NoteEditor`는 JSX 렌더링만 담당한다.
- 태그 정규화 규칙은 `normalizeTag` / `tryAddTag` 같은 **순수 함수 유틸**(`src/lib/tag.ts` 등)로 분리한다. 훅과 `TagInput` 모두 이 유틸을 호출한다.
- **`TagInput`**은 controlled component로 둔다. Props는 `value: string[]`, `onChange: (tags: string[]) => void`. 키 핸들링(Enter/쉼표/Backspace/IME)은 컴포넌트 내부, 서버 통신은 일절 하지 않는다.
- `NotesContext`는 기존 시그니처를 유지한다(`updateNote(id, Partial<Note>)`로 `tags`가 자연 통과).
- 본 결정을 향후 컴포넌트에도 동일하게 적용한다 — **표시 = 컴포넌트, 로직 = 훅, 순수 규칙 = 유틸 함수**.

**Alternatives** —

- **안 A. spec 그대로, `NoteEditor`가 `tags` state를 직접 보유.**
  거부 이유: 변경 면은 가장 작지만 `NoteEditor`가 계속 두꺼워진다. 정규화 로직이 컴포넌트에 묶여 단위 테스트가 RTL에 의존하게 되고, 입력값 표 기반의 빠른 검증이 어렵다. 합성·재사용 친화도가 낮고, 향후 동일 패턴을 다른 컴포넌트로 확장할 때마다 로직이 컴포넌트에 누적된다.
- **안 C. `NotesContext`에 `addTag` / `removeTag` 도메인 메서드 추가.**
  거부 이유: spec §6의 일괄 저장 정책과 결이 맞지 않는다. 이름은 즉시 저장처럼 들리지만 실제로는 로컬 state만 갱신하게 되어 이름–동작 괴리가 발생한다. "Context는 얇은 CRUD 통로" 라는 기존 패턴과 충돌하고, 테스트 표면적이 가장 넓어진다.

**Consequences** —

- 장점
  - 정규화가 순수 함수가 되어 입력/기대값 표 기반 단위 테스트가 가장 빠르고 명확하다. RTL/jsdom 부담 없이 수십 케이스를 회귀 검증할 수 있다.
  - `NoteEditor` JSX가 얇아져 렌더 흐름을 한눈에 파악할 수 있다.
  - `TagInput`이 controlled라서 다른 폼/컨텍스트에서도 재사용 가능하다.
  - 같은 합성 패턴(컴포넌트 + 훅 + 유틸)을 후속 작업에 일관되게 확장할 수 있다.
- 단점
  - 신규 파일이 늘어난다(`TagInput.tsx`, `useNoteEditor.ts`, `lib/tag.ts`). 기능 크기 대비 구조가 다소 무거워 보일 수 있다.
  - "언제 훅으로 빼는가"의 경계 기준이 필요하다. 규칙 없이 매번 훅으로 빼면 과추상화로 흐를 수 있다 — 본 결정은 _상태 + 저장 흐름이 동시에 얽힐 때_ 추출이라는 임계를 따른다.
  - 당분간은 비대칭이 남는다(다른 컴포넌트는 아직 훅 분리 전). 점진 적용을 전제로 한다.

### 3.2 저장 타이밍 — 저장 버튼 클릭 시 일괄 PATCH/POST

**Context** — 사용자가 chip을 추가/삭제할 때 서버에 즉시 PATCH를 날릴지, 저장 버튼을 눌렀을 때 `title`/`content`와 함께 한 번에 보낼지 결정해야 한다. 기존 `NoteEditor`는 `title`/`content` 변경을 로컬 state로 모았다가 저장 버튼에서 PATCH 1회를 보내는 패턴이다. 즉시 저장으로 가면 chip 한 번 만질 때마다 네트워크 호출이 발생하고, 부분 실패/롤백 처리 복잡도가 추가되며, "저장 버튼"이라는 명시적 액션의 의미가 약해진다.

**Decision** —

- 태그 변경은 `useNoteEditor` 훅 내부의 로컬 state에만 반영한다.
- 저장 버튼 클릭 시 `title` / `content` / `tags`를 **단일 PATCH/POST**로 묶어 전송한다.
- chip 추가/삭제는 로컬 state 갱신만 수행하며 네트워크 호출이 발생하지 않는다.
- 새 노트 생성(`isCreating`)에서도 `createNote` 호출에 `tags`를 함께 전달한다(빈 배열 허용).
- 저장 중 상태(`saving`)는 기존 패턴 그대로 훅 내부 로컬 state로 표시한다.

**Alternatives** —

- **즉시 저장 — chip 추가/삭제마다 PATCH.**
  거부 이유: 기존 패턴 위반, 네트워크 비용 증가, 부분 실패 처리 복잡, "저장 버튼" UX 의미 약화. spec §6에서 명시적으로 금지하고 있는 동작이다.
- **debounce 저장 — 마지막 변경 후 N초 뒤 자동 PATCH.**
  거부 이유: 저장 시점이 사용자 눈에 보이지 않아 예측 불가능하다. `title`/`content`와 정책이 달라지면 폼 내부 일관성이 깨진다. 현재 규모에서 자동저장은 과한 복잡도다.

**Consequences** —

- 장점
  - 기존 `title`/`content` 저장 패턴과 100% 일관 — 학습 비용 0.
  - 네트워크 호출 횟수 최소. 저장이 단일 트랜잭션처럼 다뤄져 에러 처리가 단순하다.
  - `useNoteEditor`가 모든 폼 필드를 동등하게 다루므로 훅 내부 코드가 대칭적이다.
  - e2e 시나리오가 단순하다: "입력 → 저장 클릭 → 네트워크 호출 1회".
- 단점
  - 사용자가 저장을 누르지 않고 이탈하면 chip 변경이 사라진다. 단, `title`/`content`도 동일한 동작이므로 폼 내부 비대칭은 없다.
  - 다른 디바이스/탭과의 실시간 동기화는 어차피 out of scope이라 본 결정의 단점은 아니다.
  - 향후 자동저장으로 전환하려면 `useNoteEditor` 내부 동작을 바꿔야 한다. 다만 호출부(`NoteEditor` JSX)는 그대로라 변경 영향 범위는 좁다.

## 4. Out of Scope

본 PRD의 범위에서 **명시적으로 제외**하는 항목이다. 후속 스펙에서 별도로 다룬다.

### 4.1 태그 탐색·검색·필터링

- 사이드바(`NoteList`)에 태그 chip을 표시하는 기능.
- 특정 태그를 클릭/선택했을 때 해당 태그가 달린 노트만 필터링해서 보여주는 기능.
- 태그 텍스트로 노트를 전문 검색하는 기능.
- 여러 태그를 AND/OR로 조합해 노트를 좁히는 고급 필터.
- 태그별 노트 개수 집계(`react (12)` 같은 표시).

### 4.2 태그 입력 보조 UX

- 기존 태그 풀에서 가져오는 **자동완성**(`autocomplete`/suggestion dropdown).
- 자주 쓰는 태그를 상단에 추천하는 **최근 사용 태그** 영역.
- 태그 입력 중 후보를 키보드 화살표로 탐색하는 인터랙션.
- 붙여넣기로 여러 태그를 한 번에 추가하는 동작(예: `"a, b, c"` 페이스트 → chip 3개). 현 스펙은 한 chip 씩 Enter/쉼표로 확정하는 것만 지원.

### 4.3 태그 메타데이터·꾸미기

- 태그별 **색상 지정**, **아이콘 지정**, **이모지 매핑**.
- 태그 **설명/메모** 필드.
- 태그를 즐겨찾기로 고정/핀 처리.
- chip의 시각적 카테고리화(우선순위/색상 그룹).

### 4.4 태그 관리(전역)

- 전역 **태그 목록 페이지**(모든 태그를 한 화면에서 조회).
- 태그 **rename**(예: `react` → `React.js`로 일괄 변경 후 모든 노트에 반영).
- 태그 **merge**(두 태그를 하나로 합치기).
- 태그 **삭제**(특정 태그를 사용하는 모든 노트에서 일괄 제거).
- 더 이상 어떤 노트도 사용하지 않는 **고아 태그 정리** 기능.

### 4.5 데이터 모델·백엔드

- 별도 `Tag` 엔티티/테이블/엔드포인트(`GET /tags` 등) 도입.
- 태그-노트의 **다대다 관계 모델링**(조인 테이블, 정규화).
- 태그 단위의 **권한/공유** 모델.
- 태그 **동의어/별칭**(예: `js` ≡ `javascript`) 매핑.
- 태그에 대한 서버 측 **유효성 검사·중복 검사**. (현 스펙은 클라이언트에서만 처리)

### 4.6 정렬·표시 옵션

- 노트 내 태그의 **정렬 옵션**(가나다순/최근 추가순/사용 빈도순 등). 현 스펙은 사용자가 입력한 순서를 그대로 유지한다.
- chip 영역의 **truncate**, **"+N more"** 표시, **가로 스크롤**. 현 스펙은 `flex-wrap`으로만 처리.
- 태그가 많아질 때의 가상 스크롤·페이지네이션.

### 4.7 검증·제약 강화

- **노트당 최대 태그 개수 제한**(현 스펙은 무제한).
- **금칙어 필터/욕설 검사**.
- 이모지/특수문자 **차단**(현 스펙은 허용).
- 태그 길이 상·하한의 동적 정책 변경(현 스펙은 1~20자 하드코딩).

### 4.8 분석·추천

- 태그 기반 **연관 노트 추천**.
- 사용자에게 새로운 태그를 **자동 제안**.
- 사용량·트렌드 등 태그 **분석 대시보드**.

### 4.9 동기화·협업

- 여러 사용자가 동시에 같은 노트의 태그를 편집할 때의 **충돌 해소(CRDT/OT 등)**.
- 태그 변경 이벤트의 **실시간 동기화/푸시**.
- 태그 변경에 대한 **활동 로그/감사 이력**.

### 4.10 접근성·국제화 강화 작업

- 스크린리더 전용 chip 안내문구의 추가 다듬기(기본 마크업 수준은 본 스펙 범위 안).
- 태그 입력 UI의 **다국어 라벨/플레이스홀더**.
- RTL 언어 환경에서의 chip 배치 검증.

## 5. 용어 정의

| 용어                         | 정의                                                                                                                                                               |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **태그(tag)**                | 노트에 부여하는 짧은 문자열 라벨. `Note.tags: string[]`로 저장된다. 1~20자, 대소문자 무시 기준 유일.                                                               |
| **chip**                     | 확정된 태그 1개를 시각적으로 표현하는 UI 요소. 태그 텍스트와 `✕` 삭제 버튼으로 구성된다.                                                                           |
| **TagInput**                 | chip 목록과 단일 행 input을 합친 하위 컴포넌트. 키 입력·정규화·삭제를 책임지고, 서버 통신은 하지 않는다.                                                           |
| **정규화(normalization)**    | chip으로 확정되기 직전 입력값에 적용하는 일련의 규칙(trim, 빈 문자열 거부, 길이 제한, 대소문자 무시 중복 검사).                                                    |
| **대소문자 무시 중복**       | `a.toLowerCase() === b.toLowerCase()` 비교로 동일하다고 판정되는 두 문자열. 저장 케이스는 사용자가 입력한 원본을 유지한다.                                         |
| **IME 조합 중**              | 한글·일본어·중국어 등 입력기에서 글자가 아직 확정되지 않은 상태. `e.nativeEvent.isComposing === true`로 판별하며, 이 상태의 Enter는 chip 확정으로 처리하지 않는다. |
| **일괄 저장(batch save)**    | 제목·본문·태그를 한 번의 PATCH/POST에 묶어 영속화하는 방식. chip 추가/삭제마다 즉시 서버에 반영하지 않는다.                                                        |
| **레거시 노트**              | `tags` 필드가 도입되기 전에 생성된 노트. 컴포넌트에서는 `note.tags ?? []`로 방어하고, `db.json` 마이그레이션 시 `"tags": []`를 채워 넣는다.                        |
| **빈 input에서의 Backspace** | input 값이 빈 상태에서 Backspace를 누른 동작. 마지막 chip을 삭제하는 트리거로 사용한다.                                                                            |
| **chip 확정 트리거**         | input 값을 chip으로 굳히는 키 입력. 현 스펙에서는 `Enter`와 `,`(쉼표) 두 가지. 따라서 태그 텍스트 자체에는 쉼표가 포함될 수 없다.                                  |
