# Feature 기획 워크플로우

하나의 feature를 "아이디어"에서 "개발 가능한 이슈 단위"로 변환하는 파이프라인.
이 문서는 `/feature-planner` 스킬의 근거 자료이며, 스킬 생성 시 이 내용이 SKILL.md에 그대로 반영되어야 한다.

## 전체 흐름

```
spec-original.md (초기 아이디어)
    ↓ 단계 1 — 요구사항 인터뷰
spec-fixed.md (확정된 요구사항)
    ↓ 단계 2 — PRD 작성 + 기술 결정(ADR)
prd.md (단일 기준 문서)
    ↓ 단계 3 — 이슈 분해
issues.md + GitHub Issues (실행 가능한 작업 단위)
```

---

## 단계 1: 요구사항 인터뷰

### 목적

초기 기능 정의서의 모호성을 제거하고, 엣지 케이스와 결정 사항을 확정한다.

### 입력

- `docs/features/{name}/spec-original.md` (초기 기능 정의서)
- 코드베이스 (기존 구조·패턴 참조)

### 핵심 활동

**AI를 인터뷰어로 활용한다.** 프롬프트 패턴:

```
이 정의서를 바탕으로, 코드베이스를 탐색하고 구현 전에 확인이 필요한 것들을
나에게 질문해라. 한 번에 하나씩 질문하고, 추천 방식도 이유와 함께 포함하라.
```

**질문 템플릿** (AI가 다뤄야 할 영역):

1. 이 기능의 primary user는 누구인가?
2. 최소 동작 시나리오 3개는? (가장 기본적인 사용 흐름)
3. 데이터 저장 방식 — 기존 구조와의 관계는?
4. 경계 조건 — 최대값, 빈 값, 중복, 동시성은?
5. 에러 처리 — 실패 시 사용자에게 어떻게 표시하는가?
6. 기존 UI 패턴과의 일관성 — 어떤 컴포넌트를 재사용할 수 있는가?
7. 성능 제약 — 허용 응답 시간, 데이터 크기 제한은?
8. 향후 확장 가능성 — 이 결정이 다음 feature에 영향을 주는가?

**Ubiquitous Language 정의:**

- 인터뷰 과정에서 사용되는 용어를 고정한다
- 코드·문서·대화에서 같은 단어로 같은 것을 가리킨다
- spec-fixed.md에 "용어 정의" 섹션으로 명시한다

### 산출물

- `docs/features/{name}/spec-fixed.md`

### 승인 게이트

```
[GATE] 사용자가 spec-fixed.md를 읽고 확정할 때까지 대기
```

## 단계 2: PRD 작성 + 기술 결정(ADR)

### 목적

요구사항·디자인·기술 결정·범위를 하나의 문서로 통합해 "이 기능에 대해 궁금하면 여기만 보면 된다"는 단일 기준점을 만든다.

### 입력

- `docs/features/{name}/spec-fixed.md`
- `docs/features/{name}/design.md` (존재하면 참조)
- 코드베이스 (기존 패턴 참조)

### 핵심 활동

**2-1. PRD 뼈대 생성** (기술 결정은 TODO로 비워둠)

PRD 구조:

- **개요** — 기능의 목적과 범위
- **사용자 스토리** — 사용자 관점의 흐름
- **기술 결정** — 어떤 구조 위에 올릴 것인가 (여기서 ADR 작성)
- **Out of Scope** — 이번에 하지 않을 것
- **용어 정의** — spec-fixed.md의 Ubiquitous Language 동기화

**2-2. 아키텍처 3안 제안 & 비교**

AI에게 최소 3가지 안을 제안받고, **아래 7가지 고정 기준**으로 비교한다:

| #   | 기준                 | 설명                                           |
| --- | -------------------- | ---------------------------------------------- |
| 1   | 데이터 구조          | 어떤 형태로 데이터를 저장/표현하는가           |
| 2   | API 레이어 변경지점  | 기존 API에서 무엇이 바뀌는가                   |
| 3   | 상태관리 변경지점    | Context/Store에서 무엇이 추가되는가            |
| 4   | 핵심 동작            | 이 기능의 가장 중요한 동작이 어떻게 구현되는가 |
| 5   | 컴포넌트 구조        | 어떤 컴포넌트가 추가/수정되는가                |
| 6   | 기존 패턴과의 일관성 | 코드베이스의 기존 관습과 얼마나 맞는가         |
| 7   | 테스트 용이성        | 단위 테스트, E2E 테스트 작성이 쉬운가          |

비교 결과를 표로 정리한 뒤 **사용자가 선택**한다.

```
[GATE] 사용자가 3안 중 하나를 선택할 때까지 대기
```

**2-3. 선택한 안을 ADR 4요소로 PRD "기술 결정" 섹션에 작성**

```markdown
### 기술 결정 제목

**Context** — 이 결정이 필요한 맥락. 무엇을 풀고 있는가.
**Decision** — 무엇을 선택했는가.
**Alternatives** — 거부한 안과 **각각의 거부 이유**. (이유 없는 거부 금지)
**Consequences** — 트레이드오프. **장점뿐 아니라 단점도 반드시 명시**한다.
```

**2-4. Out of Scope 작성**

"하지 않을 것"을 구체적으로 나열한다. AI에게 특히 중요한 이유:

- 명시하지 않으면 AI가 맥락을 확장 해석해 범위 초과 구현을 한다
- 예: "태그 색상 커스터마이징", "태그 자동 추천", "태그 기반 검색" 등

```
[GATE] 사용자가 Out of Scope을 읽고 확정할 때까지 대기
```

### 산출물

- `docs/features/{name}/prd.md`

## 단계 3: 이슈 분해

### 목적

PRD를 실행 가능한 작업 단위(GitHub Issues)로 변환한다. 각 이슈는 독립적으로 TDD 사이클을 돌릴 수 있어야 한다.

### 입력

- `docs/features/{name}/prd.md`
- GitHub 프로젝트 보드

### 핵심 활동

**수직 슬라이싱 원칙:**

> "이 이슈만 완료하면 사용자에게 보여줄 수 있는 동작이 있는가?"

- **Yes** → 수직 슬라이스. TDD 사이클을 돌릴 수 있다.
- **No** → 다른 이슈와 합치거나 다시 나눈다.

수평 슬라이싱(레이어별: API → Context → UI)은 금지.

마지막 이슈까지 완료해야 동작이 나오므로 TDD 사이클을 이슈 단위로 돌릴 수 없다.

**이슈 크기:**

- 반나절~하루 안에 TDD 사이클(Red→Green→Refactor) 완료 가능한 것
- 너무 크면 피드백 루프가 느려짐, 너무 작으면 PR 오버헤드가 초과

**의존성 순서:**

- 앞 이슈의 결과가 다음 이슈의 입력이 되도록 배치
- 역방향 개발 금지

**Acceptance Criteria — Given-When-Then 형식:**

```markdown
## Acceptance Criteria

- [ ] Given [사전 조건], When [행동], Then [기대 결과]
- [ ] Given [사전 조건], When [행동], Then [기대 결과]
```

각 이슈에 반드시 AC를 포함한다. AC는 다음 단계(/test-scenarios)에서 테스트 시나리오의 씨앗이 된다.

```
[GATE] 사용자가 이슈 목록을 읽고 수직 슬라이스 기준 충족을 확인할 때까지 대기
```

**GitHub 등록:**

```bash
gh issue create --title "이슈 제목" --body "설명 + AC + 의존성"
gh project item-add <PROJECT_NUMBER> --owner <OWNER> --url <ISSUE_URL>
```

### 산출물

- `docs/features/{name}/issues.md`
- GitHub Issues (각 이슈에 AC 포함)
- 프로젝트 칸반 보드 Todo 컬럼에 배치

---

## 승인 게이트 요약

| 지점        | 확인 내용                                         |
| ----------- | ------------------------------------------------- |
| 단계 1 후   | spec-fixed.md — 모호성 제거, 용어 확정            |
| 단계 2-2 후 | 3가지 안 비교 — 사용자가 안을 선택함              |
| 단계 2-4 후 | Out of Scope — 범위 확정                          |
| 단계 3 후   | 이슈 목록 — 수직 슬라이스, AC 구체성, 의존성 순서 |

이 게이트들이 없으면 AI가 판단 없이 끝까지 달려버린다. 스킬에 반드시 `[GATE]`로 명시해야 한다.
